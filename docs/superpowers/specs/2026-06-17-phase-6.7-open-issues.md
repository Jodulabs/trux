# Phase 6.7 — Open issues (live-use audit)

*Branch: `phase-6.7-mobile-ux` · Logged 2026-06-17 from live phone + desktop testing.*

The redesign landed (theme, markdown, repo names, user/assistant roles all render). These are the
issues that surfaced when actually driving it. Each has evidence, a root cause pinned to a file, and
the proposed fix. Ordered by how much they get in the way.

---

## 1. Composer is cramped — textarea + icons on one row

**Evidence:** phone screenshot — typing box is a thin sliver with the bookmark/list/attach/send
icons crowding it on the same line.

**Root cause:** `apps/frontend/src/components/Composer.tsx:138` puts the `<textarea>` and
`.composer-tools` (all four buttons) inside one `.composer-field`, and
`apps/frontend/src/index.css:472` lays that field out as `display: flex; align-items: flex-end`.
On a narrow phone the icons claim horizontal space and the textarea collapses to a sliver.

**Fix:** Restructure into two stacked rows.
- Row 1: `<textarea>` full-width, taller resting height (`min-height` ~44px), auto-grow preserved.
- Row 2: the action icons (bookmark / list / attach) left, a spacer, the round send/stop button right.
- `index.css`: `.composer-field` → `flex-direction: column`; new `.composer-actions` row
  (`display: flex; align-items: center`) with `margin-left: auto` on the send button.
- All `data-testid`s stay (`composer-input`, `send`, `interrupt`, `snippet-save`, `snippet-open`,
  `attach-image`, `file-input`).

## 2. "Pair phone" button is useless on the phone (and won't go away)

**Evidence:** user on the phone sees a "Pair phone" button mid-screen; tapping it opens a QR ("qr
code page"). On the phone you're *already* paired — there's nothing to scan.

**Root cause:** `apps/frontend/src/components/Sidebar.tsx:42` always renders the pair button +
`PairModal`. Pairing is inherently a desktop→phone handoff (big screen shows QR, phone scans it), so
on a touch client it's pure noise.

**Fix:** Show the pair button only where it's usable.
- Hide when the client is a touch/coarse-pointer device: `window.matchMedia('(pointer: coarse)').matches`.
- Also hide when there's no `tailscaleHost` + token (the QR would be "unavailable" anyway).
- Net: button appears on the desktop where you run pairing, and is gone on the phone — "go away once
  pairing done."

## 3. `pnpm start` floods the terminal with the QR

**Evidence:** terminal screenshot — `pnpm start` prints the full pairing QR + URLs, pushing the
"listening on…" line off-screen. The QR isn't wanted on every start; it reads as "qr code page still
opened."

**Root cause:** `apps/backend/src/index.ts:32` calls `printAccessBanner(config)`, and
`apps/backend/src/banner.ts` `printAccessBanner` always renders the full QR.

**Fix:** Split the banner.
- `pnpm start` (`index.ts`): print a compact 3-line banner — listening URL, Tailscale URL (no QR),
  and "run `pnpm pair` to show the pairing QR."
- `pnpm pair` (`pair.ts`): keep the full QR — that's its whole job.
- Implementation: add a `qr: boolean` param to `printAccessBanner` (or split into
  `printStartBanner` / `printPairBanner` in `banner.ts`); `index.ts` calls the no-QR variant,
  `pair.ts` the QR one.

## 4. `codex session list` errors on every session-discovery call

**Evidence:** terminal screenshot — `error: unexpected argument 'list' found` (twice), with codex
usage text. Fires whenever the New-conversation dialog probes for codex sessions.

**Root cause:** `apps/backend/src/routes.ts:47` runs `codex session list --json`. The current Codex
CLI removed that subcommand — `codex --help` now lists only `exec`, `resume`, `archive`, `fork`, …
(no `session`). The `execFileSync` throws, the `catch` returns `[]`, but codex's stderr still leaks
to the terminal.

**Fix:** Discover codex sessions from disk, mirroring the Claude path (`~/.claude/projects/`).
- Codex stores rollouts at `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`. Line 1 of each is a
  `session_meta` record: `{"type":"session_meta","payload":{"id":..., "cwd":..., "timestamp":...}}`.
- Walk that tree (or read the newest N by mtime), parse the first line, filter `payload.cwd === cwd`,
  return `{ sessionId: payload.id, updatedAt: mtimeMs }`, sort desc, slice 10.
- Drop the `execFileSync('codex', …)` call entirely — no subprocess, no stderr leak.
- Update/add a test in the routes/sessions suite with a temp `~/.codex/sessions` fixture.

## 5. Desktop browser hits a bearer-token wall with no guidance

**Evidence:** desktop screenshot — opening `http://localhost:4317` shows "A bearer token is required
to access this instance" + a bare "Paste your token" box. User: "what's a bearer token, where to get
it?"

**Root cause:** `apps/frontend/src/components/TokenGate.tsx` gives no hint where the token lives.
Auth is on (`TRUX_AUTH=1` + `TRUX_SECRET` in `~/.trux/.env`), so even the owner on the same box must
present the token, and nothing tells them where it is.

**Why not just trust loopback:** tempting to skip auth for `127.0.0.1`, but `tailscale serve`
proxies remote requests *to* `127.0.0.1:4317` — so from Fastify's view a phone-over-Tailscale request
also looks like loopback. Trusting loopback would silently disable auth for everyone. Not worth the
fragility of sniffing `X-Forwarded-For`.

**Fix (UX, recommended):** Make `TokenGate` self-explanatory for the owner.
- Add copy: "Your token is `TRUX_SECRET` in `~/.trux/.env`. On your phone, scan the QR from
  `pnpm pair` instead of typing it."
- Keep the paste box for the desktop browser case.

**Optional follow-up (not in this pass):** a `pnpm token` helper that prints just the secret, for
quick copy on the desktop.

## 6. Workspace picker is a flat dump of every directory *(low priority — deferred)*

**Evidence:** user — "workspace showing all possible list of directory is not great ux. i prefer
something better." Flagged as *not* the immediate concern.

**Root cause:** `apps/frontend/src/components/NewConversationDialog.tsx:59` is a single `<select>`
that flattens every worktree of every workspace root into one option list. Since fix-era
`workspaces.ts` now surfaces all repos one level under a non-repo root, the list grew long and
undifferentiated — just a wall of folder names, no grouping, no search, no recency.

**Direction (when picked up — not this pass):** options, roughly in effort order —
- Group options with `<optgroup>` per workspace root (cheap, keeps native `<select>`).
- Sort by most-recently-used (reuse conversation history) so the repos you actually touch float up.
- Replace the `<select>` with a searchable/typeahead picker (filter as you type) — best UX, most work.
- Let the picker remember the last-used repo and default to it.

Deferred deliberately; capture here so it isn't lost.
